# Module 7 — ML Notes

## Zia AutoML Pipeline (trained, validated, not live-served)

- **Pipeline**: `district-risk-scoring`, built via Catalyst QuickML's AutoML pipeline builder
- **Dataset**: `risk_training_data.csv`, engineered from CaseMaster (district-month buckets with rolling-average and heinous-ratio features), generated by the `/ml/training-data` endpoint and uploaded to Stratus
- **Target**: `risk_label` (High / Normal), derived from a >=1.5x rolling-3-month-average volume spike rule
- **Features**: `district`, `month_number`, `prior_month_count`, `rolling_3month_avg`, `heinous_ratio`
- **Result**: Zia's pipeline automatically selected an **XGBoost Classifier** after running mean/mode imputation, label encoding, target encoding, and Explorekit feature engineering. Training status: Success. Endpoint published and manually verified in the console tester (correct "Normal" prediction on a low-activity test input, 200 response).

## Why the live API uses rule-based scoring instead

Calling the published QuickML endpoint from within a Catalyst Function consistently returned `ORGID_HEADER_UNAVAILABLE` — a platform-level SDK/service issue where the org-context header isn't attached correctly on that specific call path, even after upgrading to the latest `zcatalyst-sdk-node`. Rather than risk the demo on an unresolved platform quirk, `/ml/risk-scores` reimplements the same feature logic (rolling 3-month average, heinous-offence ratio) as a transparent, explainable scoring function directly in code. This has a genuine advantage for a law-enforcement tool: every risk flag comes with a stated reason, rather than an opaque model score.

## If revisited post-hackathon

The likely fix is calling the QuickML endpoint's raw REST API directly (with explicit `CATALYST-ORG`, `X-QUICKML-ENDPOINT-KEY`, and OAuth headers) instead of the SDK's `quickml.predict()` wrapper, which appears to not populate the org header correctly when invoked from a Function's request context.
